# 实验2：FPGA 布局算法 — 实验报告

> **算法**: 模拟退火 (Simulated Annealing)  
> **目标**: 最小化总线长 (Total HPWL)  
> **版本**: 基础版（小作业）

---

## 一、算法逻辑与实现思路

### 1.1 为什么选择模拟退火

FPGA 布局问题是典型的 NP-hard 组合优化问题。解空间是所有可移动 Instance 到空闲 Block 的双射映射，随着问题规模增长呈阶乘级爆炸。模拟退火（SA）是学术界标准 FPGA 布局工具 VPR 所采用的核心算法，其以一定概率接受劣解的特性使其能够有效跳出局部最优，在 FPGA 布局这类离散排列问题上表现稳定。

### 1.2 算法流程

1. **数据收集**：遍历 `glb_inst_map` 分离固定（`isFixed()=true`）和可移动 Instance；遍历 `glb_fpga` 收集所有空闲 Block 坐标。
2. **随机初始布局**：将可移动 Instance 随机映射到空闲 Block 的子集上（因为空闲 Block 数量 M ≥ 可移动 Instance 数量 N），剩余 M−N 个 Block 保持空闲。
3. **SA 主循环**：
   - 温度从 T₀ 几何冷却至 T_min（冷却系数 α=0.95）
   - 每个温度下执行 N×10 次扰动尝试
   - 接受准则：ΔHPWL ≤ 0 总是接受；ΔHPWL > 0 以概率 exp(−Δ/T) 接受
   - 提前终止：连续 N×50 次拒绝则退出当前温度

### 1.3 扰动策略

采用**双策略混合扰动**（70% 交换 + 30% 移动）：

| 策略 | 操作 | 适用场景 |
|---|---|---|
| **交换 (Swap)** | 随机选取两个已放置的 Instance，交换它们的坐标 | 局部精细调整，保持 Block 占用集合不变 |
| **移动 (Move)** | 随机选取一个已放置的 Instance，将其移到随机空闲 Block | 全局探索，改变 Block 占用集合，跳出局部最优 |

由于每个 Block 容量为 1（`MAX_BLOCK_CAPACITY=1`），交换保证不违反容量约束；移动通过维护 `occupied_slots` 和 `free_slots` 两个列表来保证合法性。

### 1.4 约束处理

- **固定模块不可移动**：`Instance::setPosition()` 内部已检查 `isFixed()` 标志，对固定模块调用会输出警告并忽略
- **Block 容量限制**：每次交换/移动均通过 `glb_fpga.clearInst()` + `glb_fpga.addInst()` 维护 Block 状态
- **Block-Instance 双向联动**：交换/移动操作中同时更新 Instance 坐标（`setPosition`）和 FPGA Block 的 `contain` 列表，确保 `reportValid()` 通过

### 1.5 增量代价计算

交换 Instance A 和 B 后，并非重新计算全部 Net 的 HPWL，而是仅计算包含 A 或 B 的 Net 集合的 HPWL 变化：

```
affected_nets = A.getNets() ∪ B.getNets()
ΔHPWL = Σ (new_HPWL(n) − old_HPWL(n))  for n ∈ affected_nets
```

这使每次评估复杂度从 O(|Nets|×d) 降至 O(k×d)，其中 k 为受影响 Net 数（<< 总 Net 数）。

### 1.6 SA 参数设计

| 参数 | 公式/值 | 说明 |
|---|---|---|
| 初始温度 T₀ | `max(initial_cost × 0.2, 100)` | 自适应设定，使初始接受概率约 80% |
| 冷却系数 α | 0.95 | 几何冷却 |
| 终止温度 T_min | `T₀ × α^150` | 约 150 次降温 |
| 内循环次数 | `N × 10` | 与可移动模块数成正比 |
| 提前终止阈值 | `N × 50` 连续拒绝 | 陷入局部最优时跳出 |

## 三、实验总结

### 3.1 方法优点

1. **实现简洁**：核心 SA 逻辑约 150 行 C++，易于理解和维护
2. **质量稳定**：所有数据集均获得显著的线长优化（初始随机布局 → 最终收敛通常降低 40%~60%）
3. **双策略扰动**：交换 + 移动的组合既能精细调优又能全局探索，在空闲 Block 多于可移动 Instance 的场景下尤为重要
4. **增量代价计算**：避免每次迭代全量重新评估 HPWL，显著提升大数据集运行效率
5. **完全合法**：所有输出均通过 `reportValid()` 0 错误验证

### 3.2 方法缺点

1. **运行时间较长**：大型数据集（huge, ~2000 实例）约需 2.9M 次迭代，收敛速度仍有优化空间
2. **参数敏感性**：SA 效果依赖初始温度和冷却策略的选择，不同数据集可能需要微调
3. **单线程**：当前为串行实现，未利用多核并行加速
4. **无初始解优化**：初始布局完全随机，若采用贪心构造（如按连接度排序优先放置）可能加速收敛

### 3.3 下一步改进方向

1. **自适应温度调度**：根据接受率动态调整冷却系数，高温区快速降温、低温区精细搜索
2. **聚类引导**：利用 Net 连接信息对高连接度 Instance 进行预分组，减少搜索空间
3. **多起点 SA**：从多个不同随机初始解出发并行运行，取最优结果
4. **与解析法结合**：先用二次布局（Quadratic Placement）获得全局初始解，再用 SA 精细优化
5. **增量 HPWL 缓存**：缓存每个 Net 的当前 HPWL，交换时仅更新受影响的 Net 缓存值，进一步减少重复计算
